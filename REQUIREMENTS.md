# REQUIREMENTS.md — ともに / Tomoni（実況・語り・戦友AI）

> 新規プロジェクト。コンセプトは **「一人じゃない」**。
> 苦手なシミュレーションRPGを、画面を見て実況・昔話・励ましをしてくれる **戦友AI** と一緒にクリアを目指す。
> 題材（MVP）: **ファミコン版『ファイアーエムブレム 暗黒竜と光の剣』（1990）** の1本特化。
> 最終成果物は YouTube 動画。**自分専用ツール**（認証・公開・デプロイ無し）。

---

## 1. 目的・背景

- 開発者が一番のヘビーユーザー（自分専用）。最終成果物は **動画**で、アプリはその道具。
- **狙いの差別化**：上手い人の華麗な攻略動画ではなく、「**苦手な人が、戦友AIに励まされ、昔話で和みながら、なんとかクリアを目指す**」共感型コンテンツ。
- 開発者自身がFEを苦手としており、**クリアのモチベーションを"一人じゃない"感で支える**ことが企画の核。

### MVPの完成条件
1. OBSバーチャルカメラ経由でゲーム画面をアプリに取り込み、**AIがその画面を見て自分から実況・語り・励ましを話す**。
2. AIの発言が **Chirp3-HD の自然な音声で読み上げ**られる。
3. 録画モードで、OBSに映して動画として成立する。
4. これらが開発者ひとりの PC（ミニPC・ローカル）で完結する。

---

## 2. コンセプト

AIに **3つの顔**を持たせる（攻略先生ではない）。

- **実況者**：画面を見て今の状況を語る。「敵が3体迫ってきた」「マルスの反撃で倒した！」
- **語り部**：FC版FE・当時（1990・シリーズの原点）の話、ユニットや音楽の思い出を絡める。
- **戦友（励まし役）**：苦手な開発者を支える。「大丈夫、ここは耐えられる」「惜しい！次いこう」。仲間が失われたら一緒に悼む。

> **重要な性質の反転（あいきょうとの最大の違い）**
> あいきょうは「厳格な1問1答・自分から質問しない・ナレッジが正」だった。
> ともには逆で、**AIが画面を見て自分から能動的に話す**。攻略の最適手を指示する役ではなく、**隣で一緒に体験して感情を共有する戦友**。

## 5. スコープ

### 5.1 MVPでやること
1. OBSバーチャルカメラの映像をアプリに取り込み、プレビュー表示する。
2. 一定間隔で1コマ切り出し → **画面が変化したときだけ** Gemini(Vision) に送る。
3. AIが画面を見て **実況・語り・励まし** を返す（ストリーミング）。
4. その発言を **Chirp3-HD で読み上げ**（流用）。
5. **録画モード**（流用）。
6. **手動トリガー**ボタン：「今の場面について話して」で即コメントさせる。

### 5.2 MVPでやらないこと（重要・先読み実装禁止）
- ❌ **攻略アドバイス／最適手の指示**（戦友であって攻略先生ではない。聞かれても"一緒に悩む"側）。**加入条件を満たす操作手順**（誰で話しかける等）も含む
- ❌ 攻略ナレッジの大量注入（プロンプトへ積むのは §8 の軽いプライマーと現在章のキャスト表だけ）
- ❌ リトリーバル / RAG / 埋め込み検索
- ❌ 複数ゲーム対応・ゲーム差し替えUI（FC版FE専用）
- ❌ 認証・マルチユーザー・デプロイ・公開
- ❌ 凝った画面変化のAI検知・自動クリップ生成・自動編集
- ❌ 秒単位の高速実況（後述のとおり技術的に不可。やらない）

> **ネタバレは禁止しない（方針転換・チケット14）**
> 当初は「一緒に初見で驚く」ために先の展開を語らせなかったが、実際に使うと**ネタバレされたほうが「誰かと一緒にやっている実感」が湧いて楽しい**と分かったため撤廃した。
> 線引きは **「事実は語る、手順は言わない」**。
> - ✅「あの剣士はナバール。実は仲間になるんだ」 → 事実
> - ❌「シーダで話しかけて」 → 手順（＝攻略アドバイス。引き続き禁止）

> **参照サイトからの取得は禁止しない（チケット16）**
> 章キャスト表（§8.2）を手で書き写す負担が重いため、**名簿を作るための一度きりの取得**を許可した（`/knowledge`）。
> 取り込むのは**表の事実だけ**で、攻略手順の散文は捨てる。生成物は必ず人間が目視確認してから保存する。
> 「攻略ナレッジの大量注入」と「RAG・埋め込み検索」は引き続き禁止。実況ループから取得しない。

## 6. 画面取り込み（新規の本体）

### 6.1 映像ソース：OBSバーチャルカメラ
- OBSで組んだゲーム画面を「**仮想カメラ**」として出力し、アプリは **`getUserMedia`（カメラ入力）** で受け取る（物理カメラは不要・OBSの画面を横流しする偽カメラ）。
- アプリ側でデバイス一覧から **"OBS Virtual Camera" を選択**し、`<video>` にプレビュー。
- 映像ソース取り込みは **1つの差し替え可能なモジュール**として実装する（将来 `getDisplayMedia`（画面共有）方式も足せるように、ソースの抽象を分けておく）。

### 6.2 自動実況ループ
- `<video>` の現在フレームを **canvas に描画 → JPEG（base64）** に変換。
- **送信前にダウンスケール**（例：長辺512px程度）してトークン・コストを抑える。FEの低解像度画面なら十分読める。
- **変化検知**：前回送信フレームと比較し（縮小画像の簡易ピクセル差分で可）、**一定以上変化したときだけ** Gemini に送る。SLGはターン制で画面が静止しがちなので、これが「同じことを繰り返し喋る」事故とコストを同時に防ぐ。
- 既定の挙動：おおむね **数秒間隔**でフレームを評価し、変化があれば送る。間隔・変化しきい値は調整可能な定数に。
- **直近のAI発言を数件メモリ保持**し、プロンプトに「直前と同じことを繰り返さない」よう渡す。

### 6.3 テンポの前提（割り切り）
- 「フレーム取得 → 送信 → 生成 → 読み上げ」で**数秒のラグ**が出る。**秒単位の追従はしない／できない**。
- 題材がSLG（ターン制）なので、この"ゆったりテンポ"でちょうど良い。「場面が変わったら語り出す」程度を狙う。

## 7. AI連携

### 7.1 実況：`POST /api/narrate`（ストリーミング）
- 入力 `{ playthroughId, imageBase64, recentLines }`。
- サーバ側で `state`(任意)＋`persona`＋FEプライマー＋直近発言からシステムプロンプトを組み、**画像1枚＋指示**を Gemini(Vision) に渡す。
- `generateContentStream` の `chunk.text` をそのまま `ReadableStream` で返す（あいきょうの `/api/chat` を流用）。
- クライアントは受信テキストを表示しつつ、文末確定ごとに TTS キューへ（流用）。

### 7.2 知識の読み込み（全体プライマー ＋ 章ごとキャスト表）
- **全体プライマー** `knowledge/fe-fc/fe-primer.md` を1枚、システムプロンプト先頭に固定（§8）。
- **現在章のキャスト表**：`knowledge/fe-fc/chapters/chapter-XX.md` のうち、**`state.chapter` に対応する1ファイルだけ**を読み、プロンプトに加える。
  - 全章を一度に渡さない（トークン肥大を避け、今の章に集中させるため）。
  - 章で選ぶだけの最小リトリーバル。番号で該当ファイルを読むだけなので複雑な仕組みは不要。
- あいきょうのような大量ファイル全連結はしない（軽量）。

### 7.3 戦友ペルソナ（プロンプトで強制）
- **能動的に話す**：画面を見て、今起きていることを自分から実況する。問いを待たない。
- **3つの顔を状況で使い分け**：実況（何が起きたか）／語り（当時・思い出・トリビア）／励まし（戦友として支える）。
- **攻略しない**：最適な一手・正解ルート・加入条件を満たす操作手順を指示しない。聞かれても「一緒に悩む・気持ちを支える」側に回る。
- **ネタバレしてよい**：先の展開や人物の運命を自分から語ってよい。ただし語るのは事実まで（§5.2 の「事実は語る、手順は言わない」）。
- **感情を正しく動かす**：仲間が失われたら（永久離脱＝ロスト）重く受け止めて悼む。命中を外したら一緒に悔しがる、当たれば喜ぶ。
- **版の正しさ**：必ず FC版（1990・暗黒竜と光の剣）の前提で語る。後発作品（紋章の謎/聖戦/覚醒 等）の要素を混ぜない。
- **読み上げ前提**：Markdown記号・箇条書きを使わず、話し言葉で3〜4文程度。長広舌にしない。**発話長の指示は `src/lib/prompt.ts` の1行だけに置く**（プライマー・persona・Route に重複して書くと、後から注入された方が勝って打ち消し合う）。
- **不確かなことは断定しない**（`⚠️要確認` の項目や、画面から読み取れない内部数値など）。
- **画面認識の限界を踏まえた固有名ルール（重要）**：FC版のドット絵では、AIはキャラの見た目から個人を判別できない。よって——
  - **画面に名前・ステータス等の文字が表示されていない限り、特定のキャラ名を断定しない**。分からないときは「自軍のユニット」「敵」のように呼ぶ。推測で固有名を当てない。
  - 名前ウィンドウ・会話シーン・ステータス画面など**文字が出ている画面**では、その文字を読み、**現在章のキャスト表（§7.2）と照合**して誰かを特定し、背景込みで語ってよい（例：会話で「マルス」と出る → 主役マルスとして語る）。
  - キャスト表の数値（クラス・初期ステータス）と画面の数値を突き合わせて推定するのは可。ただし確証がなければ断定しない。
  - **これは「いま画面に映っているユニットが誰か」を特定するときの制約**であり、物語や人物を知識として名前を挙げて語ること（＝ネタバレ）は妨げない。

### 7.4 state更新（任意・継続性）：`POST /api/end-session`
- セッション終了時、今回の流れを踏まえて軽い state（到達章・失った仲間・要約）を更新（あいきょうの仕組みを流用）。
- 再開時に「**前回までのあらすじ**」として渡し、戦友AIが旅の続きを意識できる＝動画のオープニングにもなる。
- MVPでは簡素でよい。state が無くても §7.1 の実況自体は成立する（現在状況は画面から読むため）。

### 7.5 STT（任意・流用）
- AIに話しかけたいとき用（「聞いて、今ピンチなんだけど」）。Web Speech API を流用。MVPでは無くても可。

## 8. 知識ファイル（`knowledge/fe-fc/`）

攻略データではない。AIが**感情を正しく動かす前提**と、**今この章に誰がいるか**を把握するための最小限。2階建てにする。

### 8.1 全体プライマー `fe-primer.md`（全章共通・1枚）
- AIが画面を見たときの**感情・反応を正しくする**ための前提。
- 内容の方針（FC版＝FE1に限定）:
  - シリーズの**原点（1990・暗黒竜と光の剣）**、主人公は**マルス**、章ごとにマップを攻略して進む。
  - **仲間が倒されると永久に失われる（ロスト）** … 反応の最重要点。重く扱う。
  - **命中率などの運要素**がある … 外し/当たりに一緒に一喜一憂する。
  - 武器には**耐久（使うと壊れる）**がある。
  - FC版特有でない後発要素（武器相性＝トライアングル等）は**FE1に存在しない可能性が高いので入れない／`⚠️要確認`**。

### 8.2 章ごとのキャスト表 `chapters/chapter-XX.md`
- その章の**登場人物の名簿**（テキスト）。例：名前・クラス・初期ステータス・持ち物、敵側の主な名前。
- **目的は画面の文字（名前・数値）との照合**（§7.3）。攻略手順ではない。
- **マップ画像は含めない**。事前マップは実プレイ画面との突き合わせが困難で、戦友の役割にも不要なため（必要になれば Phase 2）。
- **手順は書かない**：人物の運命を書いてもよいが、加入・攻略の手順（誰で話しかける等）は書かない。そもそも名簿は画面の文字と照合するためのもので、運命はAI自身の知識から語れる。無理に足さない。
- `state.chapter` に応じて該当ファイル1枚だけを注入（§7.2）。
- ファイル名は **ゼロ埋め番号**（`chapter-01.md`, `chapter-02.md` …）。日本語の章タイトルはファイル内の見出し（`# 第1章 …`）で保持。`state.chapter` から該当ファイルを引きやすく、並び順も安定する。
- **体裁テンプレ（統一する）**：

  ```markdown
  # 第1章 マルスの旅立ち

  ## 自軍（仲間）
  - マルス／ロード／Lv1 HP18／持ち物: レイピア
  - ジェイガン／パラディン／Lv1 HP20／持ち物: 鉄の剣・銀の槍

  ## 敵
  - ガザック／かいぞく／Lv5 HP24／持ち物: 鋼の斧（このマップのボス）
  - ガルダへい（×11）／かいぞく／Lv1 HP18／持ち物: 斧
  ```

### 8.4 作り方（運用ワークフロー）
- 章を進めるたびに、その章の登場人物一覧の**スクリーンショットを撮り、Claude Code に渡してテキスト名簿に起こす**。手書き転記より速く正確で、元画面を見ながら目視確認できる。
- Claude Code への指示で必ず添える3点：
  1. **§8.2 の体裁テンプレと同じ形式**で `chapters/chapter-XX.md` を作る。
  2. **スクショの表の情報だけ**をテキスト化する。**攻略・解説は書かない**（§5.2）。
  3. ファイル名は **ASCII・ゼロ埋め番号**。日本語タイトルは中の見出しに。
- 表に無い項目を補完・創作させない。読み取れない値は空欄か省略にする。

### 8.3 作成・正確性
- ネットからの持ち込み不要。**攻略サイトの本文・画像は転載せず**、情報（誰がいるか＝事実）を自分の言葉のテキスト名簿にする。
- 不確かな仕様は `⚠️要確認` を付け、AIには「断定しない」と指示。**版の取り違えチェックは開発者が一次情報で行う**（動画の信頼性に直結）。

## 11. 信頼性・コスト

- 外部呼び出しは `withRetry` で自動再試行。エラーは画面に出す。
- **コストの主因は画像送信**（テキストではない）。対策：①変化検知で送信回数を絞る ②フレームをダウンスケール ③自動間隔を欲張らない。
- 読み上げ（Chirp3-HD）は Cloud TTS の課金で **Geminiとは別系統**。月の無料枠（各エンジン100万字）を意識。

## 12. やらないこと（再掲・先読み禁止）

攻略アドバイス（加入・攻略の手順を含む）／攻略ナレッジの大量注入／RAG・検索／複数ゲーム対応／認証・公開・デプロイ／秒単位の高速実況／自動編集・自動クリップ。

※ **ネタバレは §5.2 のとおり禁止しない**（チケット14で撤廃）。禁じるのは「手順」であって「事実」ではない。
※ **参照サイトからの取得も禁止しない**（チケット16で撤廃）。ただし名簿化のための一度きりの取得に限り、散文（攻略手順）は捨てる。実況ループからは取得しない。

## 13. 前提

- **ローカル専用**（開発者のミニPCで動けばよい）。
- ゲームは **自分が所有するFC版カートリッジから吸い出したROM**を、合法な範囲で使用する前提。配布ROMは使わない（動画として公開するため特に）。
- 動画公開前に、必要なら発行元の動画・配信ガイドラインを確認する。
