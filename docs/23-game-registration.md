# 23 ゲーム登録（画面から primer 下書きを作る）

## 概要

20 で「`knowledge/<slug>/` にファイルを置けばゲームが増える」形になるが、**primer を手書きする負担**は残る。`/knowledge` に**ゲーム登録**の流れを足し、`game.json` と `primer.md` を画面から作れるようにする（`REQUIREMENTS.md §8.4 その0`）。

**これは「AIがゲームを学習する」仕組みではない。** チケット16とまったく同じ流儀——**一度きりの取得 → 目視確認・手直し → ファイル保存**——であり、以後は保存済みファイルを毎回注入するだけ。RAG も、実況ループからの取得も、引き続き禁止（`§12`）。

## 流れ（2段入力）

1. **タイトル・機種・発売時期を入力**（例：「ファイアーエムブレム 暗黒竜と光の剣」／ファミコン／1990年）。slug は自動提案し、編集できる（`[a-z0-9-]+`）。
2. **参照URL（最大3件）** → primer の**下書きだけ**を生成。
3. **目視確認・手直し**（テキストエリアで編集できる）→ 保存で `knowledge/<slug>/game.json` と `primer.md` を書き出す。

- **①の入力は生成プロンプトに「同定のアンカー」として渡す。** 機種・発売時期を与えないと、後発リメイク・移植版の情報が混ざる（16 で Shift-JIS を UTF-8 で誤読して「GBA版」と幻覚した実測がある。文字コード判定は 16 と同じ `detectCharset` / `decodeHtml` / `htmlToText` を再利用する）。
- 生成するのは **§8.1 の型に沿った primer** ：**そのゲームの同定／感情が動く場面／基本ルール／当時の背景ネタ／そのゲームで何が「手順」にあたるか／画面認識上の固有事情**。**攻略手順の散文は捨てる。**
- **章キャスト表（16・21）は任意**。章・ステージ構造を持たないゲームは primer だけで成立する（state が空でも実況が成立するのと同じ理屈）。
- 生成される `game.json` に `knowledgeBuilder`（21 の抽出設定）は**入れない**。必要なゲームだけ後から手で足す＝そのゲームで `/knowledge` の章抽出が使えるようになる。

関連: `REQUIREMENTS.md §5.2 / §8.0 / §8.1 / §8.4 その0 / §10 / §12` ／ 依存: **20**（`game.json` と `knowledge/<slug>/` の基盤）、16（取得・文字コード・保存の機構を再利用）、21（`/knowledge` のゲーム選択UIを共有）

## 設計メモ

- **LLM に Markdown を書かせない**（16 の教訓）。構造化JSON（同定・感情ポイント・基本ルール・背景ネタ・手順の定義・画面認識の注意）だけを返させ、**§8.1 の体裁は純関数で組む**（`src/lib/primer-render.ts`）。体裁ブレとプロンプト注入の余地を消す。
- **不確かな項目には `⚠️要確認` を付けさせる**（版の取り違えは動画の信頼性に直結する。断定させない）。裏取りは開発者が一次情報で行う（`§8.3`）。
- **書き込みは save Route に集約する。** 現状 `writeFile` はリポジトリで `/api/knowledge/save` の1箇所だけ。ここを増やすなら**同じ厳格さ**で：slug をサーバ側で検証し（`[a-z0-9-]+`・`..` を弾く）、パスはサーバで組み立て、クライアントの文字列をそのまま `path.join` に渡さない。
- **既存ゲームの上書きは確認を挟む**（`fe-fc` の primer を事故で潰さない）。既に `knowledge/<slug>/` があれば警告する。

## 実装時に分かったこと（次に触る人へ）

- **モデルは `gemini-2.5-flash`・思考ON にした**（Todo の当初案 `flash-lite` から変更）。登録は**一度きりの生成**で、外すと版の取り違えがそのまま動画に出る。実況と違ってテンポ制約が無いので、`thinkingConfig: { thinkingBudget: 0 }` を付けない。1回の呼び出しなのでコストも問題にならない。
- **本文の上限は章抽出と別に持つ**（`KNOWLEDGE_PRIMER_MAX_TEXT_CHARS = 20000`）。章抽出は章ごとに分割して投げるので 300k で足りるが、登録は**全URLぶんを1回の呼び出しに載せる**。primer に要るのは同定のアンカーであって攻略ページ全文ではない。
- **`writeFile` は `/api/knowledge/save` に集約したまま**にした（新ルートを作らない）。`kind` で `chapters` / `game` を分ける。**存在チェックの向きが逆**なのが要点——章保存は「実在するゲームにしか書かない」（404）、ゲーム登録は「既にあれば止める」（409・`overwrite` で明示的に解除）。
- **`game.json` はサーバ側で組み立てる。** クライアントに生JSONを編集させると、壊れた定義や `knowledgeBuilder` の混入をそのまま受け入れてしまう。フォームの各値を個別に検証し、**空文字は落とす**（`lostLabel` を落とす＝ロストの概念が無い・§8.0）。実測でも『スーパーマリオブラザーズ』は `lostLabel` が空で返り、正しく省かれた。
- **`⚠️要確認` は純関数が付ける**（`uncertain: true` の行末）。LLM に記号を書かせない。ただし**モデルは自信過剰で、付く数は少なめ**（マリオでは1件だけ）。下書きは「確認済み」ではないという前提を UI 側の警告文で担保している。
- 検証は『スーパーマリオブラザーズ』（FC・1985）で行った。**章構造もロストの概念も無いゲーム**で、`progressLabel: "現在のステージ"` / `progressPlaceholder: "例: ワールド1-3"` が出て、実況も成立した（＝汎用化の実ゲーム検証を1本兼ねた）。**検証後に `knowledge/smb-fc/` は削除済み。**

## 完了条件

- タイトル・機種・発売時期とURL数件から、**目視確認できる primer の下書き**が出る。
- 保存すると `knowledge/<slug>/game.json` と `primer.md` ができ、**コードを触らずに**そのゲームのプレイスルーを作れる（20 の受け入れ条件と地続き）。
- 生成物に攻略手順の散文が入っていない。不確かな箇所に `⚠️要確認` が付く。
- 既存ゲームを誤って上書きしない。

## 注意

- **生成された primer をそのまま信じない。** LLM が書いた「そのゲームの事実」は、版の取り違えを含みうる。**目視確認と手直しを挟む前提の道具**であって、自動化ではない（16 と同じ）。
- **primer に攻略ナレッジを積まない。** 「感情を正しくする最小限」という §8 の性格を、生成の便利さを口実に太らせない。**大量注入の禁止は維持**（`§12`）。
- 参照先の利用規約・`robots.txt` を確認する。取得は**一度きり**。
